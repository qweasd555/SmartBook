# 智记后端接口文档

## 1. 文档说明

### 1.1 目标
本文档面向 `Spring Boot + MySQL` 后端开发与前端联调，定义"智记"App 的核心接口协议、鉴权方式、数据结构和业务约束。

### 1.2 适用范围
- V1.0：登录鉴权、OCR 识别、记账、账单列表、统计报表、预算、21 天挑战、用户中心
- V1.1：共享账单、储蓄目标

### 1.3 架构原则
- 前后端分离，RESTful 风格
- 统一返回体、统一错误码
- 账单是核心事实表，报表和挑战数据由账单实时聚合
- OCR 识别与正式记账分离，识别结果不直接入账

### 1.4 环境约定
- 开发环境：`http://localhost:8080/api`
- 生产环境：`https://api.zhiji.com/api`
- 编码：`UTF-8`，协议：`HTTPS`
- 时间格式：`yyyy-MM-dd HH:mm:ss`，时区：`Asia/Shanghai`
- 金额单位：元，数据库 `decimal(12,2)`

## 2. 通用接口规范

### 2.1 请求头

```http
Content-Type: application/json
Authorization: Bearer {token}
X-App-Version: 1.0.0
```

登录和发送验证码接口无需 `Authorization`。

### 2.2 统一返回体

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

所有接口响应均包裹在此结构中，`data` 为具体业务数据，失败时 `data` 为 `null`。

### 2.3 返回码约定

| code | 含义 |
|------|------|
| 0 | 成功 |
| 40001 | 参数错误 |
| 40002 | 业务校验失败 |
| 40101 | 未登录或 token 失效 |
| 40301 | 无权限 |
| 40401 | 资源不存在 |
| 50001 | 系统异常 |
| 50002 | 外部服务异常（如 OCR 服务） |

### 2.4 分页约定

```json
{
  "pageNo": 1,
  "pageSize": 20,
  "total": 156,
  "list": []
}
```

> status：0=正常 1=禁用

## 6. OCR 与图片上传接口

### 6.1 上传图片并发起识别
`POST /ocr/recognitions`

> 请求格式为 `multipart/form-data`。后端收到图片后：存至本地、转 Base64 调用 OCR 服务、结果写入 ocr_record，返回 taskId。前端拿到 taskId 后轮询查询结果。

#### 请求参数

| 参数 | 类型 | 说明 |
|------|------|------|
| file | File | 图片文件，仅支持 jpg/png，限制 10MB |

#### 响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": "ocr_123"
  }
}
```

### 6.2 查询 OCR 识别结果
`GET /ocr/recognitions/{taskId}`

> 前端轮询此接口，建议间隔 1 秒。status=1 时展示识别结果供用户确认，status=3 时引导手动录入。

#### 响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "taskId": "ocr_123",
    "status": 1,
    "recognized": true,
    "confidence": 0.93,
    "amount": 18.00,
    "merchantName": "瑞幸咖啡",
    "occurredAt": "2026-04-08 14:30:00",
    "categorySuggestion": {
      "categoryName": "餐饮"
    },
    "rawText": "瑞幸咖啡 生椰拿铁 18.00"
  }
}
```

> status：0=处理中 1=就绪 2=已确认入账 3=识别失败

### 6.3 OCR 结果确认入账
`POST /ocr/recognitions/{taskId}/confirm`

> 用户在确认页可修改识别结果，提交后才正式写入 transaction，ocrTaskId 自动关联。

#### 请求参数

```json
{
  "type": 0,
  "amount": 18.00,
  "categoryName": "餐饮",
  "merchantName": "瑞幸咖啡",
  "occurredAt": "2026-04-08 14:30:00",
  "remark": "生椰拿铁"
}
```

#### 响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "transactionId": 30001
  }
}
```

## 7. 账单接口

### 7.1 手动创建账单
`POST /transactions`

```json
{
  "type": 0,
  "amount": 25.50,
  "categoryName": "交通",
  "merchantName": "滴滴出行",
  "occurredAt": "2026-04-08 12:15:00",
  "remark": "打车回家"
}
```

> type：0=支出 1=收入  
> categoryName 枚举值：餐饮 / 交通 / 购物 / 娱乐 / 医疗 / 收入 / 其他

### 7.2 编辑账单
`PUT /transactions/{transactionId}`

> 请求体同 7.1，传需要修改的字段即可。

### 7.3 删除账单
`DELETE /transactions/{transactionId}`

> 逻辑删除，status 置为 1，不影响历史统计。

### 7.4 获取账单详情
`GET /transactions/{transactionId}`

#### 响应

{

## 9. 预算接口

### 9.1 设置预算
`POST /budgets`

> 同一用户同分类同周期只能有一条预算，重复提交则覆盖更新。

```json
{
  "periodType": "WEEK",
  "categoryName": "娱乐",
  "amount": 200.00,
  "alertThreshold": 80
}
```

> periodType：WEEK / MONTH  
> categoryName 为空时表示设置总预算  
> alertThreshold：预警阈值（%），默认 80

#### 响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 40001
  }
}
```

### 9.2 删除预算
`DELETE /budgets/{budgetId}`

#### 响应

```json
{
  "code": 0,
  "message": "success",
  "data": null
}
```

## 14. 业务规则

### 14.1 账单
- type：0=支出 1=收入，金额必须大于 0
- 删除为逻辑删除（status=1），不影响历史统计
- ocrTaskId 不为 null 即表示来自 OCR 识别

### 14.2 OCR
- 识别结果仅作为候选，用户确认后才入账
- status=3 时前端引导用户手动录入，原始图片保留显示

### 14.3 预算
- 同用户同分类同周期唯一，重复 POST 覆盖更新
- usedAmount 实时从 transaction 聚合，不存库
- 使用率达到 alertThreshold 时推送一次预警，不重复提醒

### 14.4 挑战
- 首次记账自动开启挑战
- 当天有至少 1 笔 status=0 的账单即视为完成，自动写入 challenge_day
- 中断不清零历史记录

### 14.5 共享账单
- 结算金额动态计算，不预存每人应付结果
- 支持平均分摊和自定义分摊

## 15. 安全建议

- JWT，token 有效期 30 天
- 上传文件仅允许图片格式，限制 10MB
- 所有资源接口校验归属（账单、预算、目标只能操作自己的）
- 短信验证码：60 秒限发一次，同号码每天最多 10 次
- OCR 接口限流，防止滥用
